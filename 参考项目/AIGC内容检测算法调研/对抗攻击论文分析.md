## 二、ADAT 任务定义

### 任务目标

**Adversarial Detection Attack on AI-Text (ADAT)**：

> 对 AI 生成的文本进行轻微扰动，使其被检测器误判为人类写作

### 攻击设置

| 设置 | 攻击者可获得的信息 |
|------|-------------------|
| **白盒攻击** | 模型参数、梯度、训练数据 |
| **黑盒攻击** | 仅检测结果（人类/AI） |
| **动态攻击** | 检测器持续更新 |

---

## 三、HMGC 框架

### 框架概述

**Humanizing Machine-Generated Content (HMGC)** 是首个系统化的对抗攻击框架：

```
攻击流程：

1. 训练替代模型
   ↓（黑盒设置）
   
2. 词重要性排序
   ↓（梯度 + 困惑度）
   
3. 掩码词替换
   ↓（BERT 生成候选）
   
4. 后检查约束
   ↓（语义保持）
   
5. 攻击成功？
   ↓
   是 → 输出对抗文本
   否 → 回到步骤 2
```

## 四、核心方法

### 4.1 词重要性排序

**双因素排序**：

$$I_{w_i} = (1-\alpha) I^{g}_{w_i} + \alpha I^{p}_{w_i}$$

**因素 1：梯度重要性**
$$I^{g}_{w_i} = \left\| \frac{\partial L_\theta}{\partial e_{w_i}} \right\|$$

**因素 2：困惑度重要性**
$$I^{p}_{w_i} = \text{ppl}(t \setminus w_i) - \text{ppl}(t)$$

**关键洞察**：
- 梯度大的词 → 对检测结果影响大
- 移除后困惑度增大的词 → 关键词

### 4.2 掩码词替换

```python
# 替换流程
for word in sorted_words_by_importance:
    # 1. 掩码关键位置
    masked_text = text.replace(word, "[MASK]")
    
    # 2. 用 BERT 生成候选
    candidates = bert_mlm.predict(masked_text, top_k=10)
    
    # 3. 贪婪搜索最佳替换
    for candidate in candidates:
        new_text = text.replace(word, candidate)
        if detector(new_text) < threshold:
            return new_text  # 攻击成功
```

### 4.3 三大约束

| 约束 | 目的 | 实现方式 |
|------|------|----------|
| **词性约束** | 保持语法正确 | 替换词必须同词性 |
| **扰动比例约束** | 限制改动量 | 最多替换 X% 的词 |
| **USE 约束** | 保持语义一致 | USE 相似度 > 阈值 |

## 七、关键洞察

### 7.1 困惑度是核心

**发现**：
```
AI 文本 → 低困惑度（确定性高）
人类写作 → 高困惑度（多样性高）
对抗攻击 → 人为增加困惑度 → 伪装成功
```

**验证实验**：

| 文本类型 | 困惑度 | 检测结果 |
|---------|--------|---------|
| AI 原文 | 15 | AI (95%) |
| 对抗文本 | 35 | 人类 (90%) |
| 人类写作 | 40 | 人类 (95%) |

### 7.2 攻击为什么有效？

**根本原因**：
1. **检测器过拟合**：依赖统计特征（困惑度、词频）
2. **特征可操控**：这些特征容易被改变
3. **语义不变性**：轻微改动不影响语义

## 八、对你的警示

### 8.1 单一检测器的脆弱性

| 防御方案 | 正常文本 | 对抗文本 |
|---------|---------|---------|
| 单一检测器 | 95% | **20%** |
| 集成多模型 | 92% | **40%** |
| 对抗训练 | 88% | **75%** |

### 8.2 推荐防御策略

```
多层防御架构：

Layer 1: Fast-DetectGPT（零样本检测）
  ↓ 可被绕过
  
Layer 2: 监督分类器（RoBERTa）
  ↓ 可被绕过
  
Layer 3: 文体特征（人工设计）
  ↓ 难以绕过
  
Layer 4: 对抗训练（持续更新）
  ↓ 提升鲁棒性
  
Layer 5: 多模型投票
  → 最终判断
```

## 九、实践建议

### 9.1 构建鲁棒检测系统

**必须做到**：
1. **集成多种方法**（零样本 + 监督 + 规则）
2. **定期对抗训练**（每月更新）
3. **监控攻击模式**（收集新攻击样本）
4. **人工审核**（高风险场景）

### 9.2 检测器更新策略

```
每周：
- 收集被绕过的样本
- 分析攻击模式

每月：
- 用新样本对抗训练
- 更新模型参数

每季度：
- 评估整体鲁棒性
- 考虑架构升级
```
